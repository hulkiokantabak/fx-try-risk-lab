Methodology · Version 2

A forecast is only useful when its event, evidence and failures are visible.

FX TRY Risk Lab estimates horizon-specific Turkish lira depreciation risk from an empirical USD/TRY history. It publishes the precise event, a frozen data cutoff, uncertainty, source condition and out-of-sample diagnostics alongside the result.

1. Event specification 2. Forecast mechanism 3. Validation 4. Data integrity 5. Expert overlay

Language rule

Validation status is explicit.

Every model output remains on a percent probability scale. Until every published horizon passes the validation gate, the interface calls it an experimental probability estimate, not a validated forecast probability.

1 · Event specification

Exactly what is being estimated

TRY depreciation is represented by an increase in USD/TRY. For each horizon, the event occurs when the future derived ECB USD/TRY reference rate is greater than or equal to the baseline rate multiplied by one plus the horizon threshold. The baseline is the latest common-date ECB EUR/TRY and EUR/USD observation available at publication.

The baseline is a daily reference rate, not an executable dealer quote. The aligned series contains only dates on which both required ECB observations exist. In the formula, t is the baseline aligned observation and t + N is exactly the Nth later aligned ECB trading observation—not an approximate calendar date.

2 · Forecast mechanism

Empirical, regime-conditioned and leakage-aware

The statistical core uses derived ECB USD/TRY history. It estimates the historical event rate among observations resembling the current market regime, then shrinks that conditional rate toward the as-of historical event rate. The regime is described by four features available at the forecast date:

  • 5-session USD/TRY percentage change;
  • 20-session USD/TRY percentage change;
  • 20-session annualized realized volatility; and
  • short-trend acceleration relative to the 20-session trend.

If a narrowly matched regime has too few examples, the conditioning is relaxed in declared steps. Bayesian shrinkage limits extreme estimates from small groups. A local reliability adjustment can then use only earlier, already resolved walk-forward forecasts near the current raw estimate.

What does not enter the probability

Current global rates, volatility indices, CBRT policy and reserves, peer currencies, and headlines remain contextual lenses. They appear in the evidence desk and expert reasoning, but do not alter the statistical probability until point-in-time histories with reliable release timestamps can be tested without look-ahead bias.

Training discipline

Only resolved labels enter

At every historical forecast date, a training example is admitted only after its full target window has ended. This keeps future outcomes out of both model fitting and probability adjustment.

Uncertainty

An interval, not false precision

The published 90% interval is a shrinkage-adjusted Bernoulli sampling interval based on the effective conditional and calibration sample. It does not cover model misspecification, regime breaks, data revisions or geopolitical discontinuities.

3 · Validation

Walk forward, compare with a baseline, preserve the live record

Backtesting uses a strict expanding window. Evaluation forecasts are separated according to the target horizon, and every training set is purged of outcomes that would not yet have resolved. The public diagnostics are reported separately for each horizon.

A horizon passes the published gate only when its walk-forward sample has at least 50 forecasts, Brier skill versus expanding climatology is positive, and expected calibration error is at most 0.100. The model-level gate passes only when every horizon passes.

Backtest versus live performance

Backtest results are research evidence, not live outcomes. Each newly issued forecast is recorded in an append-only ledger with its forecast ID, model version, data cutoff, baseline, horizon estimates and uncertainty. Outcomes are attached only when their target observations later become available. The interface reports live forecasts issued and live horizon outcomes resolved separately from each horizon's backtest N.

4 · Data integrity

Semantic validation before fallback

Each source adapter checks structure, minimum observation count, plausible values and observation age. A response that is empty, malformed, implausible or too old fails even if the server returned HTTP success. Failed payloads never replace a valid last-good cache.

A cached source is usable only if it independently passes the same validation and age gate. Otherwise that input is marked unavailable; missing data are not converted into a neutral score. The public snapshot reports source status, latest observation, age, cache use and technical warnings. If either ECB series required for derived USD/TRY is unusable, the quantitative forecast build is blocked.

5 · Expert overlay

Structured judgment stays separate from the statistical forecast

When published, the Atlas, Bosphorus, Flow and Vega specialist views receive the same frozen evidence pack. Their house view, confidence and dissent are displayed as an overlay, not silently blended into the statistical result. This separation preserves both model auditability and meaningful disagreement.

  • Atlas examines global macro and broad-dollar conditions.
  • Bosphorus examines Turkish policy, inflation credibility and reserves.
  • Flow examines spot, peers, positioning proxies and market microstructure.
  • Vega examines volatility, tails and event asymmetry.

6 · Limits and appropriate use

A disciplined monitor, not an oracle

  • Historical associations are not causal and can fail after a regime change.
  • Long-horizon outcomes overlap, even with reduced evaluation cadence.
  • The reference-rate event may differ from an executable market price or investor return.
  • Sampling intervals do not quantify geopolitical shocks, capital controls or structural breaks.
  • Contextual lenses may be stale or unavailable; the source-health panel should be read first.
  • No output is investment advice, a guarantee or an automated trading instruction.
